RAG 基础概念完全指南

一、RAG 是什么？

定义

RAG（Retrieval-Augmented Generation，检索增强生成） = 检索 + 生成

为什么需要 RAG？

大模型的缺陷 RAG 的解决
知识过时（训练数据截止） 检索最新数据
不知道私有数据 检索企业内部知识库
会产生幻觉（编造信息） 基于检索到的事实回答
无法解释回答来源 可以追溯引用来源

RAG vs 微调 vs 提示工程

方法 成本 更新频率 适用场景
提示工程 最低 实时 简单任务
RAG 低 实时 知识密集型任务
微调 高 需重新训练 特定领域、风格定制

三、核心概念详解

1. 向量（Vector/Embedding）

定义： 将文本转换成一组数值，表示文本的语义特征

核心思想： 语义相似的文本，向量在空间中距离更近

常用 Embedding 模型：

模型 维度 特点
OpenAI text-embedding-3-small 1536 通用性好
OpenAI text-embedding-3-large 3072 效果更好
BGE（中文） 768 中文效果好，开源免费
M3E（中文） 768 中文效果好，开源免费

2. 向量数据库（Vector Database）

定义： 专门存储和检索向量的数据库

核心能力：

  • 向量存储
  • 相似度检索（找到最相近的K个向量）
  • 元数据过滤

主流选择：

数据库 特点 适用场景
Chroma 轻量、易上手 开发测试、小规模
Milvus 高性能、分布式 企业级、大规模
Pinecone 云托管、免运维 快速上线
Weaviate 功能丰富、支持混合检索 复杂场景
Qdrant 高性能、开源 自部署

3. 切片（Chunking）

定义： 将长文档拆分成小块，便于检索

为什么需要切片？

  • 模型有 Token 限制，不能一次处理整本书
  • 小块检索更精准，减少噪声

切片策略：

策略 原理 优点 缺点
固定长度 每 N 个字符切一次 简单 可能切断语义
递归切片 按段落→句子→字符递归切 保持语义结构 需要调整参数
语义切片 用模型判断语义边界切 语义完整 计算成本高
按文档结构 按标题、章节切 结构清晰 需要文档有结构

关键参数：

  • Chunk Size（切片大小）： 通常 300-1000 tokens
  • Chunk Overlap（重叠大小）： 通常 10-20%，避免语义断裂

4. 相似度计算

三种常见方式：

方式 公式 特点
余弦相似度 cos(θ) = A·B / ( A
欧氏距离 √(Σ(aᵢ-bᵢ)²) 关注绝对距离
点积 A·B = Σaᵢbᵢ 简单快速

实际使用： 大多数向量库默认用余弦相似度

5. Top-K 检索

定义： 召回相似度最高的 K 个文档片段

K 怎么选？

  • K 太小：可能漏掉相关信息
  • K 太大：引入噪声，Token 消耗增加
  • 通常选择 3-10

四、进阶概念

1. 混合检索（Hybrid Search）

问题： 纯向量检索可能漏掉精确匹配

解决方案： 向量检索 + 关键词检索，结果融合

常用方法： RRF（Reciprocal Rank Fusion）

2. 重排（Reranking）

问题： 向量检索的第一名不一定是最相关的

解决方案： 用更精确的模型对召回结果重新排序

为什么需要重排？

  • 向量检索用 Bi-Encoder（快速但不够精确）
  • 重排用 Cross-Encoder（慢但更精确）

常用重排模型：

  • BGE-Reranker（中文）
  • Cohere Rerank（API）
  • Jina Reranker

3. 路由（Routing）

问题： 不是所有问题都需要检索知识库

解决方案： 先判断问题类型，再决定走哪条路

路由类型：

  • 闲聊 → 直接回答
  • 知识问答 → RAG 检索
  • 工具调用 → 执行工具
  • 复杂任务 → Agent 规划

4. 元数据过滤

定义： 检索时按元数据筛选文档

常见元数据：

  • 时间（创建时间、更新时间）
  • 来源（部门、文档类型）
  • 权限（公开、内部、机密）
  • 作者

五、RAG 效果评估

评估指标

指标 含义 计算方式
召回率（Recall） 相关文档被找出来的比例 找到的相关文档 / 总相关文档
精确率（Precision） 找出的文档中有多少是相关的 找到的相关文档 / 找到的总文档
MRR 第一个正确答案的排名 1 / 正确答案排名
NDCG 考虑排序位置的指标 复杂公式

评估框架

RAGAS： 自动评估 RAG 系统

  • Faithfulness（忠实度）：回答是否基于检索内容
  • Answer Relevance（答案相关性）：回答是否切题
  • Context Relevance（上下文相关性）：检索内容是否相关

六、常见问题与解决

问题1：召回不准

原因：

  • 切片策略不合理
  • Embedding 模型不适合
  • Top-K 太小

解决：

  • 优化切片大小和重叠
  • 尝试不同的 Embedding 模型
  • 增加混合检索
  • 加入重排

问题2：回答有幻觉

原因：

  • 检索内容与问题不相关
  • 模型自己编造

解决：

  • 优化检索策略
  • 提示词约束：”只根据提供的内容回答”
  • 要求标注来源

问题3：回答太慢

原因：

  • 检索慢
  • 模型生成慢
  • 上下文太长

解决：

  • 向量库索引优化
  • 减少召回数量
  • 使用更快的模型
  • 缓存热门问题

问题4：知识库更新

场景： 文档内容变化后，如何更新向量库？

方案：

  • 全量重建：简单但慢
  • 增量更新：只更新变化的文档
  • 双写机制：写入数据库同时写入向量库
