LLM Prompt Caching 与 KV Cache 深度解析

摘要：本文从 Transformer 的 KV Cache 原理出发，深入分析 Anthropic Prompt Caching 的设计机制，包括缓存键计算、缓存断点策略、global/org scope 的区别，以及 Claude Code 的实际实现代码。帮助读者理解 LLM 推理优化的核心技术。

一、KV Cache：Transformer 推理的核心优化

1.1 什么是 KV Cache

Transformer 模型在推理时，每一层都会计算 Query (Q)、Key (K)、Value (V) 三个向量。其中 K 和 V 可以被缓存起来，避免重复计算。

注意力计算公式：

当模型生成 token 时，每个新 token 都需要与之前所有 token 计算注意力。如果没有 KV Cache，每次生成都要重新计算所有历史 token 的 K 和 V，这会造成巨大的计算浪费。

1.2 KV Cache 的工作原理

步骤 操作 说明
1 输入 token 序列 [token_1, token_2, token_3]
2 计算 K、V 每个 token 生成对应的 K_i, V_i
3 缓存 K、V 存储到内存中
4 生成新 token 只需计算新 token 的 K、V，复用缓存的

核心特性：KV Cache 必须是连续的前缀

由于位置编码（如 RoPE）被”烘焙”进 K 和 V 向量中，缓存的位置信息是固定的。这意味着：

  • ✓ 可以缓存 [token_1, token_2, token_3]
  • ✗ 不能只缓存 [token_3]（位置不匹配）
  • ✗ 不能缓存非连续片段

1.3 为什么这很重要

KV Cache 是现代 LLM 推理的基石：

  • 减少计算量：历史内容无需重复计算
  • 降低延迟：生成速度更快
  • 节省成本：API 费用更低（缓存命中的 token 更便宜）

2.1 从 KV Cache 到 Prompt Caching

Prompt Caching 是 Anthropic 在 API 层面提供的缓存服务，本质上是让用户”预约” KV Cache 的存储和复用。

2.3 多个 cache_control 的处理

一个请求可以有多个 cache_control，每个对应一个”检查点”：

匹配策略：从后往前，贪婪最长匹配

  1. 先检查 key_3（最长）
  2. 如果命中 → 全部复用，结束
  3. 如果未命中 → 检查 key_2
  4. 以此类推，找到最长命中

2.4 API 响应解读

字段 含义
cache_read_input_tokens 从缓存命中的 token 数（便宜）
cache_creation_input_tokens 新创建缓存的 token 数（较贵）
input_tokens 未缓存的普通输入 token

两个值可以同时非零，说明缓存是分段独立管理的。

三、Global vs Org Scope：缓存池的层级

3.1 缓存池的概念

缓存池是服务端内存中存储 KV Cache 的区域，不同 scope 对应不同的缓存池：

Scope 缓存池范围 适用场景
global 全球所有用户共享 静态内容，所有用户相同
org 同一账号内共享 用户特定内容
null 不缓存 动态内容

3.2 为什么 MCP 工具禁用 Global Scope

MCP（Model Context Protocol）工具是用户自定义的外部工具，每个用户的配置不同：

如果使用 global scope：

  • 用户 A 的缓存 key 包含 mcp__github__repo
  • 用户 B 的缓存 key 包含 mcp__jira__issue
  • 缓存键不同，无法共享

解决方案：有 MCP 工具时，降级为 org scope

这样每个用户有自己的缓存池，不会互相污染。

3.3 对单个用户的实际影响

对于单个用户来说，global 和 org scope 的效果相同：

  • 第一次请求：创建缓存
  • 后续请求：命中缓存

区别只在于”跨用户共享”这个服务端优化层面。

五、服务端实现：协议与自由

5.1 API 协议是双向契约

方向 必须遵守的内容
客户端 → 服务端 cache_control 位置、scope 值、请求格式
服务端 → 客户端 cache_read_input_tokens、cache_creation_input_tokens、响应格式

5.2 服务端的实现自由

API 协议规定”做什么”，服务端自由决定”怎么做”：

协议规定 服务端自由实现
cache_control 放在末尾 具体存储结构（N 个条目？增量存储？）
返回缓存命中统计 匹配算法（从前往后？从后往前？）
scope 决定缓存池 淘汰策略（LRU? LFU? TTL?）

不同服务提供商可以有完全不同的内部实现，只要符合协议即可。

总结

  • KV Cache 是 Transformer 推理优化的核心，必须是连续前缀，不能缓存中间片段
  • Prompt Caching 是 API 层面的缓存服务，通过 cache_control 标记预约缓存
  • 缓存键由内容哈希计算，任何差异都会导致未命中
  • 多个 cache_control 对应多个检查点，服务端从后往前贪婪匹配
  • global vs org scope 决定缓存池范围，对单用户体验相同
  • Claude Code 只在最后一条消息加 cache_control，assistant 消息不加
  • API 协议是双向契约，服务端内部实现有自由度

理解这些机制，有助于优化 API 使用成本，设计更高效的 LLM 应用。
