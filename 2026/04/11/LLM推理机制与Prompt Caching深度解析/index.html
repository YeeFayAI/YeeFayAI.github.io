LLM推理机制与Prompt Caching深度解析

摘要：本文深入探讨LLM推理过程中的KV Cache技术原理，解析Prompt Caching API的设计思想与使用场景，并对比流式输出与传统输出方式的差异。理解这些底层机制，有助于Agent开发者更好地优化应用性能与成本。

一、KV Cache：Transformer推理的核心优化

什么是KV Cache

在Transformer架构中，Attention机制需要计算Query、Key、Value三个矩阵。KV Cache的核心思想是：缓存已计算过的Key和Value矩阵，避免重复计算。

推理过程的两个阶段

LLM推理分为两个阶段，每个阶段都依赖KV Cache：

阶段 处理内容 KV Cache行为
预填充 处理输入prompt 一次性计算所有token的K/V并缓存
生成 输出每个新token 边输出边计算边缓存

预填充阶段：模型将整个输入prompt的所有token一次性处理，每个token计算其K/V矩阵并存入缓存。这一步是并行的，速度很快。

生成阶段：模型逐个生成新token。每生成一个token，都需要查询前面所有token的K/V（复用缓存），计算当前token的K/V，存入缓存，然后输出。

为什么生成阶段必须用KV Cache

假设生成第100个输出token时：

  • 没有缓存：需要重新计算前99个token的K/V矩阵 → 计算量爆炸
  • 有缓存：直接查表获取前99个token的K/V → 高效

这就是为什么KV Cache是Transformer推理的基本优化，每次推理都在使用。

二、Prompt Caching：跨请求的缓存复用

两层KV Cache的区别

层次 作用范围 存在条件
请求内KV Cache 单次请求内 始终存在，推理引擎自动使用
跨请求KV Cache 跨多个请求 需要Prompt Caching功能支持

没有Prompt Caching时，每次请求结束后KV Cache就被丢弃，下次请求需要重新计算。

Prompt Caching的本质

Prompt Caching = 将KV Cache的能力通过API暴露给开发者

什么时候应该使用缓存

缓存的价值取决于三个条件：

条件 说明
内容稳定 多次请求中内容不变
内容较大 token数量足够多（通常>1000）
会复用 短时间内会被多次使用

System Prompt最适合缓存：内容稳定、通常较大、每次请求都会发送。

Messages通常不适合缓存：每轮对话都在变化，缓存命中率低。

缓存能节省什么

层面 能否节省 说明
网络传输 ❌ 不能 每次请求必须发送完整内容
服务器计算 ✅ 能 命中缓存时复用KV矩阵，减少90%成本
响应延迟 ✅ 能 跳过预填充阶段，响应更快

三、流式输出：优化用户感知延迟

传统方式 vs 流式输出

传统非流式：服务器生成完整响应后一次性返回。用户等待整个生成过程结束才能看到内容。

流式输出：每生成一个token就立即返回给客户端。用户几乎立刻开始看到内容。

四、Agent开发者需要掌握的层次

作为Agent开发者，不需要深入理解推理引擎的实现细节，但需要了解API层面的优化技巧：

必须掌握

  • Agent架构设计（循环、状态、消息流）
  • 工具系统（注册、分发、权限）
  • Prompt组装与缓存策略
  • API使用技巧（Streaming, Tool Use, Caching）

了解概念即可

  • KV Cache是什么，解决什么问题
  • Prompt Caching怎么用，什么时候用
  • 为什么Streaming能降低感知延迟

不需要学习

  • Continuous Batching实现细节
  • PagedAttention内存管理
  • Flash Attention CUDA优化
  • 模型训练（预训练、SFT、RLHF）

总结

  • KV Cache是Transformer推理的基本优化，每次推理都在使用，分为预填充和生成两个阶段
  • Prompt Caching是API功能，让开发者能利用跨请求的KV Cache复用，节省服务器计算成本
  • 流式输出优化用户感知延迟，让用户更快看到响应内容
  • 网络传输无法通过缓存节省，每次请求必须发送完整内容
  • Agent开发者应聚焦API使用技巧，推理引擎优化是厂商的责任

理解这些机制，能帮助Agent开发者做出更好的架构决策和性能优化。
