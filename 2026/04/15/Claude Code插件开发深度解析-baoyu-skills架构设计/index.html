Claude Code插件开发深度解析 - baoyu-skills架构设计

主要内容：深入解析 baoyu-skills 项目的架构设计，包括 Plugin 与 Skill 的层级关系、目录结构规范、配置系统优先级、Bun 运行时机制以及 Skill 之间的调用关系。

最近在研究 baoyu-skills 项目时，我发现它是一个相当精巧的设计。初看可能会有些困惑——Plugin 和 Skill 有什么区别？配置文件该放哪里？脚本是怎么运行的？这些问题的答案其实藏在项目的架构设计里。今天我想把自己理解到的底层逻辑整理出来，希望能帮到同样在探索这个项目的同学。

Plugin 与 Skill：两个层级的关系

我一开始最困惑的就是这个问题：Plugin 和 Skill 到底是什么关系？

简单来说，Plugin 是容器，Skill 是功能单元。一个 Plugin 可以包含多个 Skill。你可以把 Plugin 理解成一个 npm 包或一个项目仓库，而 Skill 则是这个仓库里具体的功能模块。

baoyu-skills 这个 Plugin 里就包含了多个 Skill，比如 article-illustrator、imagine、translate 等。当你在 Claude Code 里启用这个 Plugin 时，它里面的所有 Skill 都会被加载进来。你不需要一个个去启用 Skill，Plugin 层级的管理已经帮你做好了这件事。

这种设计的好处是：你可以把相关的功能打包在一起，方便分发和管理。比如这个 baoyu-skills Plugin 就把图像生成相关的功能都整合在了一起。

Skill 的目录结构：约定优于配置

每个 Skill 都有一个固定的目录结构，这种约定让一切都变得可预测：

SKILL.md 是最关键的文件。它告诉 Claude 这个 Skill 是做什么的、怎么用、有哪些参数。当用户触发这个 Skill 时，Claude 会先读取 SKILL.md 来理解上下文。写好这个文件，就等于定义了 Skill 的”接口协议”。

scripts/ 目录存放实际的执行逻辑。有意思的是，这里的脚本都是 TypeScript 文件，不需要编译成 JavaScript。为什么能这样？后面我会专门解释。

references/ 目录则用来放一些参考文档，比如 API 文档、使用示例等，帮助 Claude 更好地理解 Skill 的使用场景。

配置系统：多层级的优先级

配置文件该放哪里？这个问题我问了好几遍才搞清楚。

baoyu-skills 的配置系统有多个层级，优先级从高到低：

优先级 配置来源 说明
1 (最高) CLI 参数 命令行直接指定的参数
2 EXTEND.md Skill 目录下的扩展配置
3 环境变量 系统级的环境变量
4 项目级 .env .baoyu-skills/.env
5 用户级 .env ~/.baoyu-skills/.env
6 (最低) 默认值 脚本中的内置默认值

这个设计很实用。比如你有一个 API Key 是所有 Skill 共用的，就放在用户级的 .env 里；如果某个项目需要特殊的配置，就在项目目录下建一个 .env 文件覆盖。

需要注意的是，这些 .env 文件需要你自己创建。项目不会自动生成，你需要手动创建目录和文件。

运行时机制：Bun 直接运行 TypeScript

“为什么代码能直接跑？”这可能是很多人第一次看到项目时的疑问。

答案在于 Bun。Bun 是一个现代的 JavaScript 运行时，它原生支持 TypeScript。这意味着你不需要预先编译 .ts 文件，Bun 可以直接执行它们。

baoyu-skills 项目充分利用了这个特性。所有的脚本都是用 TypeScript 写的，运行时通过 Bun 直接执行。这大大简化了开发流程——写代码、保存、直接运行，没有中间的编译步骤。

当你调用一个 Skill 时，Claude Code 会根据 SKILL.md 里的定义，找到对应的脚本并用 Bun 执行。SKILL.md 里会明确写出执行方式：

其中 ${BUN_X} 会根据你的环境自动替换为 bun 或 npx -y bun。

Skill 之间的调用关系

baoyu-skills 里的 Skill 不是孤立存在的，它们之间会有调用关系。

举个例子，article-illustrator 这个 Skill 的作用是为文章生成配图。但实际生成图片的逻辑，它调用了另一个 Skill —— imagine。这种设计体现了 Unix 哲学：每个 Skill 做好一件事，复杂的功能通过组合来实现。

调用关系是单向的，不存在循环：

当你使用 article-illustrator 时，你可能意识不到 imagine 在背后工作。这种封装让用户体验更加简洁，同时也让代码更加复用。imagine 可以被其他需要生成图片的 Skill 调用，而不需要重复实现。

理解这种调用关系很重要，因为它帮你搞清楚了每个 Skill 的职责边界。当你需要自定义或扩展某个功能时，就知道该在哪个层级下手。

共享代码的分发机制

多个 Skill 之间难免会有共享的代码，比如工具函数、类型定义等。baoyu-skills 用了一个有意思的方式来处理这个问题。

项目根目录有一个 packages/ 目录，存放共享的代码。然后通过同步脚本，这些代码会被复制到各个 Skill 的 vendor/ 目录下。

为什么这样做，而不是直接引用 packages/ 目录？这是为了让每个 Skill 保持自包含。当 Skill 被执行时，它需要的所有依赖都在自己的目录里，不需要去解析外部的路径。这种设计在运行时更加可靠，也方便单独发布到 ClawHub。

方案 优点 缺点
复制到 vendor/ Skill 独立、可单独发布 代码重复
引用根目录 packages/ 不重复 Skill 不能独立运行

这个项目选择了”复制”方案，兼顾了独立性和可维护性。

本地执行的本质

最后一个想聊的点，是”脚本在哪里运行”这个问题。

答案是：在你本地的机器上。

当你在 Claude Code 里触发一个 Skill 时，相关的脚本是在你本地的环境里执行的，而不是在某个远程服务器上。这意味着：

  • 脚本可以访问你本地的文件系统
  • 可以调用你本地安装的工具和程序
  • 执行结果直接返回给 Claude Code

这也是为什么你需要安装 Bun，以及为什么配置文件要放在本地项目目录里。整个过程都是本地的，Claude Code 只是一个协调者，负责理解你的意图并调用相应的脚本。

在本地运行 在云端运行
TypeScript 脚本 Google Gemini API
读取 .env 配置 OpenAI API
解析参数 图像生成模型
保存图片文件

理解这一点很重要，因为它解释了为什么某些 Skill 需要你预先安装依赖，或者为什么脚本可以操作你电脑上的文件。这不是黑盒魔法，而是本地工具链的有序协作。

总结

折腾完这个项目，我的几点心得：

  • 层级分明：Plugin 作为容器管理多个 Skill，Skill 作为功能单元独立运作。这种分层让项目结构清晰，也方便功能的组织

  • 约定优先：统一的目录结构、配置优先级、命名规范，这些都降低了理解成本。你只要熟悉一个 Skill，就能举一反三理解其他 Skill

  • 本地优先：Bun 运行时、本地执行、自包含的依赖，这些设计让整个系统简单可控。没有复杂的远程调用，一切都是你本地环境里的事情

  • 组合复用：Skill 之间可以相互调用，共享代码通过 vendor 分发。这让功能可以组合，代码可以复用，避免了重复建设

如果你打算基于 baoyu-skills 开发自己的 Skill，理解这些底层逻辑会让你的工作事半功倍。记住：写好 SKILL.md 定义接口，把逻辑放在 scripts/ 里，必要时利用 references/ 补充文档。配置文件跟着 Skill 走，优先级从局部到全局。这套规则一旦熟悉，开发起来就会很顺手。
