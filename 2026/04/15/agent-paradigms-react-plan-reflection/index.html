智能体三大经典范式详解

摘要：ReAct、Plan-and-Solve、Reflection 是构建智能体的三种经典范式。本文深入分析每种范式的工作原理、适用场景与设计权衡，帮助开发者根据实际需求选择合适的架构方案。

为何需要不同的范式

大语言模型虽然强大，但在处理复杂任务时面临挑战：如何在推理与行动之间取得平衡？如何避免在多步骤任务中迷失方向？如何提升初始解决方案的质量？三种经典范式分别从不同角度回应这些问题。

ReAct：推理与行动的协同

核心思想

ReAct（Reasoning and Acting）由 Shunyu Yao 于 2022 年提出，其核心思想是将”推理”与”行动”显式结合，形成”思考-行动-观察”的循环。

在 ReAct 诞生之前，主流方法分为两类：”纯思考”型（如思维链）能进行复杂推理但无法与外部世界交互；”纯行动”型直接输出动作但缺乏规划和纠错能力。ReAct 认识到思考与行动是相辅相成的——思考指导行动，行动结果反过来修正思考。

工作流程

每个时间步，智能体执行以下循环：

  • Thought：智能体的”内心独白”，分析当前情况、分解任务、规划下一步
  • Action：决定采取的具体动作，如调用工具或输出最终答案
  • Observation：执行 Action 后从外部返回的结果

这个循环不断重复，将新的观察追加到历史记录中，形成不断增长的上下文，直到智能体认为已找到最终答案。

适用场景

场景类型 具体例子 为何适合 ReAct
需要外部知识 查询实时天气、新闻、股价 可动态调用搜索工具
需要精确计算 复杂数学问题 将计算交给专用工具
需要 API 交互 操作数据库、调用服务 灵活的外部接口调用

优势与局限

优势：

  • 高可解释性：Thought 链清晰展示推理过程
  • 动态纠错能力：根据 Observation 实时调整策略
  • 工具协同能力：LLM 负责推理，工具负责执行

局限：

  • 对 LLM 能力强依赖：模型推理和格式遵循能力直接影响成功率
  • 执行效率问题：多步串行调用带来延迟和成本
  • 可能陷入局部最优：缺乏全局规划，可能在错误路径上持续投入

Plan-and-Solve：先规划后执行

核心思想

Plan-and-Solve 将任务处理明确分为两个阶段：先规划（Plan），后执行（Solve）。如果说 ReAct 像侦探根据线索一步步推理，Plan-and-Solve 则像建筑师在动工前先绘制完整蓝图。

工作流程

规划阶段：智能体接收完整问题，首先将问题分解为清晰的步骤计划。这个计划本身就是一次 LLM 调用的产物，要求输出结构化格式（如 Python 列表）。

执行阶段：智能体严格按照计划逐一执行。每一步执行都可能是一次独立的 LLM 调用，或对上一步结果的加工处理。

  • 多步数学应用题：需要先列出计算步骤再逐一求解
  • 报告撰写：先规划结构（引言、数据来源、总结）再填充内容
  • 代码生成：先构思函数和模块结构再逐一实现

与 ReAct 的对比

维度 ReAct Plan-and-Solve
决策模式 反应式，走一步看一步 预见式，先谋后动
规划能力 动态调整，但缺乏全局视野 全局规划，目标一致性强
灵活性 高，可根据观察实时转向 低，计划一旦生成就相对固定
适用任务 探索性、需要外部输入的任务 结构性强、逻辑路径确定的任务

局限性

主要局限在于计划的”静态性”——如果执行中发现某步骤无法完成，缺乏动态重规划机制。这要求规划阶段的质量足够高，或者需要设计额外的异常处理逻辑。

Reflection：自我反思与迭代优化

核心思想

Reflection 机制为智能体引入”事后自我校正循环”，使其能够审视自己的工作，发现不足，并进行迭代优化。灵感来源于人类的学习过程：完成初稿后校对，解出数学题后验算。

工作流程

三步循环：执行 → 反思 → 优化

执行：使用 ReAct 或 Plan-and-Solve 等方法生成初步解决方案

反思：调用独立的 LLM 实例扮演”评审员”，从多个维度评估：

  • 事实性错误：是否与已知事实相悖
  • 逻辑漏洞：推理过程是否连贯
  • 效率问题：是否有更优路径
  • 遗漏信息：是否忽略关键约束

优化：将初稿和反馈作为新上下文，再次调用 LLM 进行修正

适用场景

  • 代码生成与优化：从简单实现迭代到高效算法
  • 学术论文写作：多维度反思提升质量
  • 关键决策支持：需要高可靠性的场景

成本收益分析

成本：

  • 模型调用开销增加：每轮迭代至少额外两次调用
  • 任务延迟显著提高：串行过程延长总耗时
  • 提示工程复杂度上升：需为不同阶段设计专门提示词

收益：

  • 解决方案质量的跃迁：从”合格”到”优秀”
  • 鲁棒性与可靠性增强：内部纠错机制

结论：适合对结果质量要求极高、对实时性要求相对宽松的场景。

范式选择策略

任务特征 推荐范式 理由
需要外部信息检索 ReAct 动态调用工具，实时获取数据
多步骤结构化推理 Plan-and-Solve 全局规划，避免迷失
对结果质量要求极高 Reflection 迭代优化，持续提升
快速响应优先 ReAct 或 Plan-and-Solve 避免 Reflection 的多轮迭代
复杂综合任务 组合使用 如 Plan-and-Solve + Reflection

总结

  • ReAct 将推理与行动结合，适合需要动态交互的任务，优势在于灵活性和可解释性
  • Plan-and-Solve 先规划后执行，适合结构化推理任务，优势在于目标一致性和稳定性
  • Reflection 引入自我校正机制，适合高要求场景，以成本换取质量提升
  • 三种范式并非互斥，可根据任务特征组合使用
  • 选择范式时需权衡实时性要求、结果质量要求和任务复杂度
